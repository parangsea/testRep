# 07. 평가와 관측

하네스는 비결정적 시스템이다. **측정하지 않으면 개선할 수 없고, 관측하지 않으면 디버깅할 수 없다.**

## 평가 (Evaluation)

### 왜 eval인가
프롬프트 한 줄, 도구 하나를 바꿔도 동작이 달라진다. **회귀를 잡고 개선을 입증**하려면 반복 가능한 평가 집합이 필요하다.

### eval 설계

1. **대표 작업 집합**: 실제 사용 분포를 반영한 입력 모음(쉬움/보통/어려움/경계 사례)
2. **명확한 성공 기준**:
   - **검증 가능(programmatic)**: 테스트 통과, 정확한 값 일치, 스키마 부합 → 자동 채점
   - **판단 필요(LLM-as-judge / 사람)**: 품질·톤·유용성 → 루브릭 기반 채점
3. **결정적 채점 우선**: 가능하면 자동·재현 가능한 지표를 쓰고, 주관 채점은 보조로
4. **회귀 게이트**: 변경 전후 점수를 비교, 하락 시 차단

### 지표 예시
- 작업 성공률(task success rate)
- 평균 스텝 수 / 토큰 / 비용 / 지연
- 도구 호출 정확도(잘못된 도구·인자 비율)
- 환각·오류율, 안전 위반율

## 관측 (Observability)

### 무엇을 남기는가

```
세션
└─ 턴
   ├─ 입력 컨텍스트(토큰 수, 캐시 적중)
   ├─ 모델 출력(사고/행동)
   ├─ 도구 호출(이름, 인자, 결과, 지연)
   └─ 종료 사유
```

- **트레이스**: 한 작업의 전체 인과 사슬을 재구성 가능하게 — 어떤 컨텍스트에서 어떤 판단·행동이 나왔는지
- **구조적 로그**: 사후 집계·필터가 가능하도록 구조화(자유 텍스트 로그보다 우월)
- **재현 정보**: 입력·시드·버전을 함께 기록해 실패를 재현

### 디버깅 루프
1. 실패 트레이스 확보 → 2. 어느 단계에서 갈라졌는지 격리 → 3. 가설 수립(컨텍스트? 도구? 프롬프트?) → 4. 최소 재현 → 5. 수정 후 eval로 회귀 확인

## 검증을 하네스에 내장

평가는 오프라인 전용이 아니다. **런타임 자기검증**을 루프에 넣어 품질을 끌어올린다.

- 행동 후 결과를 모델이 관측·교정(예: 코드 수정 후 테스트 실행 → 실패면 재시도)
- 별도 검증 에이전트(작성자와 분리된 컨텍스트)로 자기승인 편향 방지 → [05. 오케스트레이션](./05-orchestration.md)
- "완료" 주장 전 **증거 수집**: 테스트 통과 로그, 산출물 확인

## 안티패턴

- ❌ eval 없이 프롬프트를 감으로 튜닝(회귀를 못 봄)
- ❌ 모델이 "다 됐다"고 한 말을 검증 없이 신뢰
- ❌ 같은 컨텍스트에서 작성자가 자기 결과를 승인
- ❌ 재현 불가능한 로그(시드·버전·입력 누락)
