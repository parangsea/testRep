# 04. 프롬프트 / 시스템 프롬프트

시스템 프롬프트는 에이전트의 **헌법**입니다. 역할·원칙·제약·출력 계약을 정의하며,
거의 모든 호출에 포함되므로 토큰 효율과 명료성이 동시에 중요합니다.

## 시스템 프롬프트의 구성

```
1. 정체성/역할      — "너는 무엇이며 무엇을 위해 존재하는가"
2. 핵심 원칙        — 우선순위, 트레이드오프 판단 기준
3. 능력/도구 안내   — 사용 가능한 도구와 사용 시점
4. 제약/금지        — 하지 말아야 할 것, 안전 경계
5. 출력 계약        — 형식, 톤, 길이, 구조
6. 환경 정보        — OS, 경로, 시간 등 실행 맥락
```

## 작성 원칙

### 명료성 > 장황함
모델은 모순·중복에 약하다. 한 가지를 한 곳에서 명확히 말하라. 같은 지시를 여러 번 다르게 쓰면 충돌한다.

### 지시의 위계를 분명히
"반드시(MUST)", "권장(SHOULD)", "가능(MAY)"을 구분하라. 모든 것을 "반드시"로 쓰면 진짜 중요한 게 묻힌다.

### 부정보다 긍정 지시
"X 하지 마라"보다 "대신 Y 하라"가 행동을 더 잘 유도한다. 가능하면 바람직한 행동을 직접 기술.

### 이유를 함께
"왜"를 곁들이면 모델이 미지의 상황에 **일반화**한다. 규칙 나열만으로는 경계 사례에서 무너진다.

## 예시(few-shot)의 힘

규칙을 길게 설명하는 것보다 **좋은 예시 1~2개**가 더 효과적인 경우가 많다.

- 입력→출력 쌍으로 형식과 톤을 동시에 전달
- 경계 사례(까다로운 입력)를 예시로 보여주면 일반 규칙보다 강력
- 단, 예시는 토큰을 쓴다 → 가장 헷갈리는 지점에만 집중 투입

## 출력 계약 (Structured Output)

후속 코드가 모델 출력을 파싱해야 한다면, **느슨한 자연어 파싱에 의존하지 말라.**

- **스키마 강제**: 도구/함수 호출로 구조화 출력을 받으면 검증을 호출 계층에서 처리 → 불일치 시 모델이 재시도
- JSON 스키마, 열거형, 필수 필드를 명시
- 출력 형식을 시스템 프롬프트와 도구 정의 양쪽에서 일관되게 유지

## 사고 유도 (Reasoning)

- 복잡한 작업은 **계획→실행** 분리가 품질을 높인다. "먼저 계획을 세우고 실행하라."
- 명시적 사고(스크래치패드/thinking)를 허용하면 다단계 추론 정확도가 오른다.
- 단, 모든 작업에 장황한 사고를 강요하면 비용·지연이 늘고 단순 작업이 느려진다 → 작업 난도에 맞춰라.

## 톤·페르소나

- 일관된 페르소나는 사용자 경험을 안정화한다.
- 단, 페르소나가 정확성·안전 지시를 덮어쓰지 않게 위계를 분명히.

## 안티패턴

- ❌ 모든 지시를 "반드시"로 도배 → 우선순위 소실
- ❌ 모순된 지시를 여러 곳에 분산
- ❌ 자연어 출력을 정규식으로 파싱(취약) — 구조화 출력을 쓸 것
- ❌ 거대한 프롬프트에 일회성 예외를 계속 누적 → 비대화·캐시 비효율
