# 02. 컨텍스트 관리

컨텍스트 윈도우는 에이전트의 **작업 기억(working memory)** 이자 가장 희소한 자원입니다.
하네스 엔지니어링에서 가장 큰 품질 차이가 여기서 발생합니다.

## 핵심 원칙: 컨텍스트는 예산이다

매 모델 호출에 들어가는 토큰을 **의도적으로 편성**해야 합니다. "일단 다 넣자"는 비용·지연·품질 모두에서 손해입니다.

```
[ 시스템 프롬프트 ] + [ 도구 정의 ] + [ 관련 지식/검색 ] + [ 대화 히스토리 ] + [ 현재 작업 ]
        고정 비용         고정 비용        선별 주입          압축 대상         가변
```

## 컨텍스트 부패 (Context Rot)

컨텍스트가 길어질수록 모델 성능이 떨어지는 현상.

- **중간 소실(lost in the middle)**: 긴 입력의 중앙부 정보가 양 끝보다 덜 활용됨
- **노이즈 누적**: 무관한 도구 출력·실패 로그가 쌓이면 판단을 흐림
- **지시 희석**: 핵심 지시가 방대한 히스토리에 묻힘

> 대응: 길이를 늘리기 전에 **신호 대 잡음비**를 높여라. 무관한 내용을 넣지 않는 것이 더 넣는 것보다 낫다.

## 주요 기법

### 1. 컴팩션 / 요약 (Compaction)

히스토리가 한계에 가까워지면 과거 대화를 **요약본으로 치환**하고 최근 맥락은 원문 유지.

- 무엇을 보존할지 명시: 결정 사항, 미해결 과제, 핵심 사실, 사용자 제약
- 무엇을 버릴지: 중간 도구 출력 원문, 탐색 과정의 막다른 길
- 요약 자체도 비용 → 한계에 임박했을 때만 트리거

### 2. 검색 기반 주입 (Retrieval / RAG)

모든 지식을 컨텍스트에 상주시키지 말고, **필요할 때 관련 조각만** 가져온다.

- 임베딩 검색, 키워드 검색, 구조적 조회(파일 트리, 심볼)
- 코드 에이전트에서는 "전체 파일 읽기"보다 "필요한 심볼/라인만" 읽는 것이 효율적
- 검색 결과는 출처와 함께 주입해 모델이 신뢰도를 판단하게 함

### 3. 프롬프트 캐싱 (Prompt Caching)

변하지 않는 접두부(시스템 프롬프트, 도구 정의, 고정 컨텍스트)를 캐시해 **반복 호출 비용·지연을 절감**.

- 캐시는 보통 **접두부 일치**로 동작 → 안정적인 내용을 앞쪽에, 가변 내용을 뒤쪽에 배치
- TTL이 있음(예: 수 분) → 주기적 호출이면 캐시 적중, 오래 쉬면 미스
- 설계 함의: 컨텍스트 순서를 캐시 친화적으로 고정하라

### 4. 구조적 컨텍스트

자유 텍스트보다 **구조화된 형태**가 모델이 파싱·활용하기 쉽다.

- 명확한 구분자/섹션 헤더, XML·마크다운 태그
- 도구 결과는 일관된 포맷으로 (03 도구 설계 참고)
- 메타데이터(파일 경로, 라인 번호, 타임스탬프)를 붙여 후속 행동의 근거로 활용

## 컨텍스트 윈도우 예산 편성 예시

```
총 200K 토큰 윈도우 가정
├─ 시스템 프롬프트 + 도구 정의   ~8K   (고정, 캐시)
├─ 프로젝트/도메인 지식          ~12K  (선별 주입, 캐시)
├─ 검색 결과(현재 작업 관련)     ~20K  (가변)
├─ 대화 히스토리                 ~나머지 (컴팩션 대상)
└─ 출력 여유분                   충분히 확보
```

## 안티패턴

- ❌ 모든 파일을 통째로 읽어 컨텍스트에 적재
- ❌ 실패한 시도의 장황한 스택트레이스를 정리 없이 누적
- ❌ 가변 내용을 앞에, 고정 내용을 뒤에 둬서 캐시를 깨뜨림
- ❌ "혹시 몰라서" 넣는 무관한 배경 정보
