# 06. 메모리와 상태

컨텍스트 윈도우는 휘발성 작업 기억입니다. 턴·세션을 넘어 정보를 보존하려면
**명시적 메모리·상태 레이어**가 필요합니다.

## 메모리의 층위

| 층 | 수명 | 예 |
| --- | --- | --- |
| **작업 기억** | 현재 컨텍스트 윈도우 | 진행 중 대화, 직전 도구 결과 |
| **단기 상태** | 세션 | 현재 작업 목록, 임시 노트, 진행률 |
| **장기 메모리** | 세션 간 영속 | 사용자 선호, 프로젝트 제약, 학습된 사실 |
| **외부 지식** | 영구/공유 | 코드베이스, 문서, 위키 |

## 무엇을 저장할 것인가

저장은 **선별**이다. 모든 것을 적으면 회상 시 잡음이 된다.

- ✅ **비자명한 사실**: 코드/히스토리에서 유추 불가능한 결정·제약·선호
- ✅ **반복 참조될 정보**: 매번 다시 알아내면 비싼 것
- ❌ **저장하지 말 것**: 코드 구조, 과거 수정, git 히스토리(이미 기록됨), 이번 대화에만 의미 있는 것

> 누가 "이걸 기억해"라고 했는데 그게 이미 코드/히스토리에 있다면, **무엇이 비자명했는지**를 물어 그것만 저장하라.

## 파일 기반 메모리 패턴

에이전트 하네스에서 흔한 구현은 **파일 시스템을 메모리로** 쓰는 것이다.

```
memory/
├── INDEX.md           # 한 줄 요약 인덱스(매 세션 로드)
├── user-preferences.md
├── project-constraints.md
└── learned-facts.md
```

- **인덱스 분리**: 전체를 매번 로드하지 말고, 한 줄 요약 인덱스만 상시 로드 → 관련될 때만 본문 회상
- **한 파일 = 한 사실**: 갱신·삭제가 쉬움, 충돌 방지
- **메타데이터**: 종류(선호/피드백/프로젝트/참조), 작성 시점 → 회상 시 신뢰도 판단
- **상대 날짜 절대화**: "내일"이 아니라 "2026-06-03"로 저장(나중에 의미가 변함)

## 메모리 위생 (Hygiene)

- **중복 방지**: 저장 전 같은 내용이 있는지 확인 → 있으면 갱신, 새로 만들지 말 것
- **오류 삭제**: 틀린 것으로 밝혀진 메모리는 제거
- **시점 고정**: 회상된 메모리는 "작성 당시 사실"이다. 파일/플래그를 가리키면 **지금도 유효한지 검증** 후 사용
- **배경 vs 지시 구분**: 회상된 메모리는 배경 정보지 새 명령이 아니다

## 상태 영속화

진행 중인 멀티스텝 작업은 중단·재개를 위해 상태를 외부화한다.

- **작업 목록/체크리스트**: 완료/진행/대기 상태를 파일로 유지 → 재개 시 위치 복원
- **저널링**: 각 스텝의 입력·결과를 기록 → 재실행 시 캐시 적중(같은 입력은 재계산 불필요)
- **결정적 재개**: 재개 시 동일 입력이면 동일 결과가 나오도록 비결정성(난수·시각) 격리

## 안티패턴

- ❌ 모든 대화를 통째로 장기 메모리에 덤프
- ❌ 코드·git에 이미 있는 정보를 메모리에 중복 저장
- ❌ 상대 시간("어제", "다음 주")을 그대로 저장
- ❌ 회상된 옛 메모리를 검증 없이 현재 사실로 단정
