# 09. 디버깅 가능한 하네스 시스템 (설계 + 구현)

지속적 개발과 검증을 1급 시민으로 두는 하네스 시스템의 설계와, 이 저장소에 포함된
참조 구현(`harness/`)의 사용법을 정리합니다.

> 이 문서는 설계 청사진이고, 실제 동작하는 최소 구현은 저장소 루트의 [`harness/`](../../harness/README.md) 에 있습니다.

## 설계 불변식

| 불변식 | 의미 |
| --- | --- |
| **재현성 (Replayable)** | 실패한 실행을 동일 입력·시드·버전으로 재실행할 수 있다 |
| **평가 게이트 (Eval-gated)** | 프롬프트/도구/모델 변경은 eval 회귀가 없을 때만 통과한다 |
| **증거 기반 완료 (Evidence-based)** | "완료"는 검증 산출물(테스트·점수)로만 인정한다 |

## 아키텍처

```
실행 레이어 ── 계측 후크 ──▶ Trace 수집기 ──▶ JSONL 저장소
                                              │
                       ┌──────────────────────┼───────────────────┐
                       ▼                       ▼                   ▼
                   디버거(replay)         Eval 하네스          메트릭/요약
                                              │ 게이트
                                              ▼
                                        CI / 머지 차단
```

두 개의 맞물린 루프:
- **이너 루프(초 단위)**: 코드/프롬프트 수정 → 단일 실행 → trace replay 로 디버깅
- **아우터 루프(분 단위)**: 변경 → 전체 eval → 회귀 델타 → 게이트 → 머지

## 핵심 컴포넌트

### 1) Trace 수집기 — 디버깅의 토대
모든 턴의 인과 사슬을 구조적으로 기록한다. 디버깅의 90%는 다음 세 필드가 좌우한다.

- `input.tokens` / `input.cacheHit` — 컨텍스트 예산·부패·캐시 디버깅
- `decision` — 모델이 *왜* 그 행동을 했는가
- `repro` — `{ seed, harnessVersion, promptHash }` 재현 키

이벤트 스키마는 [`harness/trace/schema.mjs`](../../harness/trace/schema.mjs) 참고.

### 2) 결정적 리플레이
- 실패 `runId` 를 같은 입력·시드·버전으로 재실행
- **부분 리플레이**: N턴까지 캐시 결과로 통과 후 의심 지점부터 라이브 → 이분 탐색으로 발산 지점 격리
- 비결정성(난수·시각)은 trace에서 주입(코드 내부에서 `Date.now()` 직접 호출 금지)

### 3) Eval 하네스 — 지속적 검증
- 골든 태스크(쉬움/보통/어려움/경계) + 채점기(결정적 우선, LLM-judge 보조)
- baseline 스냅샷 대비 **회귀 게이트**: 하락 시 CI 차단
- 구현: [`harness/eval/`](../../harness/README.md)

### 4) 런타임 자기검증
- 행동 후 결과 관측 → 자기 교정 (예: 코드 수정 → 테스트 → 실패면 재시도)
- **작성자와 분리된 검증 에이전트**로 자기승인 편향 차단 → [05. 오케스트레이션](./05-orchestration.md)
- "완료" 전 증거 수집 강제 → [07. 평가와 관측](./07-evaluation-and-observability.md)

## 표준 디버깅 워크플로

```
1. 실패 trace 확보 (runId)
2. timeline 재구성 → 어느 턴/도구에서 갈라졌나
3. 가설 분류: 컨텍스트? 도구? 프롬프트? 모델?
4. 부분 리플레이로 발산 지점 이분 탐색
5. 최소 재현 케이스 → evals 에 회귀 케이스로 추가   ← 핵심
6. 수정 → 해당 케이스 + 전체 eval 통과 확인
```

> 모든 버그는 고치기 전에 eval 케이스가 된다. 같은 버그가 다시 새지 않게 만드는 장치.

## 단계별 로드맵

| 단계 | 산출물 | 목표 | 상태 |
| --- | --- | --- | --- |
| P1 관측 | Trace 수집기 + JSONL + timeline 뷰어 | 보이게 | ✅ `harness/trace/` |
| P2 리플레이 | 결정적/부분 리플레이 | 재현 | 🔜 (스키마의 `repro` 준비됨) |
| P3 Eval | 골든셋 + 결정적 채점 + baseline 게이트 | 측정·차단 | ✅ `harness/eval/` |
| P4 게이트+검증 | CI 회귀 게이트 + 런타임 검증 에이전트 | 입증 | 🔜 (`runner.mjs` 종료코드로 게이트 가능) |

## 이 저장소와의 격리

하네스 툴링은 앱 빌드에 영향을 주지 않도록 분리되어 있다.

- `harness/**` 는 `.mjs` (의존성·빌드 불필요, `node` 로 직접 실행)
- 앱의 `tsc -b`(`src`만), vitest(`src/**`만), eslint(`**/*.{ts,tsx}`만) 어디에도 포함되지 않음
- 따라서 하네스 변경이 앱 빌드/테스트를 깨뜨리지 않는다
