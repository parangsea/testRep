# 01. 기초와 에이전트 루프

## 하네스란

**하네스(harness)** 는 LLM을 실제 작업에 투입하기 위해 모델 호출 주위에 두는 모든 코드·구성·절차의 총합입니다.

```
사용자 의도
   │
   ▼
┌─────────────────────────────────────────────┐
│                  하네스                       │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐  │
│  │ 컨텍스트  │→ │   모델    │→ │   도구    │  │
│  │  구성     │   │  (LLM)   │   │   실행    │  │
│  └──────────┘   └──────────┘   └──────────┘  │
│        ▲                            │         │
│        └──────── 관측·검증 ─────────┘         │
└─────────────────────────────────────────────┘
   │
   ▼
결과 + 부수효과(파일 변경, API 호출 등)
```

## 모델 vs 하네스

| | 모델 | 하네스 |
| --- | --- | --- |
| 역할 | 추론·생성 | 그 추론을 현실 작업으로 연결 |
| 바꾸기 | 비싸고 느림(재학습/교체) | 즉시 반복 가능(코드) |
| 개선 레버 | 한정적 | **대부분의 실전 개선이 여기서 나옴** |

같은 모델이라도 하네스가 나쁘면 결과가 나쁩니다. **단기적으로 가장 큰 개선 레버는 거의 항상 하네스 쪽**입니다.

## 구성 요소

1. **컨텍스트 빌더** — 매 호출마다 모델에 무엇을 보여줄지 조립 (시스템 프롬프트, 히스토리, 검색 결과, 도구 정의)
2. **모델 인터페이스** — 호출, 스트리밍, 재시도, 토큰/비용 관리, 캐싱
3. **도구 레이어** — 모델이 호출할 수 있는 함수(파일 IO, 셸, 검색, API)와 그 결과를 모델이 읽을 형태로 변환
4. **제어 루프** — 관측→판단→행동→검증의 반복과 종료 조건
5. **메모리/상태** — 턴·세션을 넘는 영속 정보
6. **가드레일** — 권한, 확인, 샌드박스, 위험 차단
7. **관측** — 로깅, 트레이싱, eval

## 에이전트 제어 루프

가장 기본이 되는 패턴은 **ReAct(Reason + Act)** 계열의 루프입니다.

```
loop:
    1. 관측(Observe)   — 현재 상태/직전 도구 결과를 컨텍스트에 반영
    2. 판단(Reason)    — 모델이 다음 행동을 결정 (계획/사고)
    3. 행동(Act)       — 도구 호출 또는 최종 답변
    4. 검증(Verify)    — 행동 결과가 목표에 부합하는지 확인
    until 종료 조건(목표 달성 / 한계 도달 / 사용자 중단)
```

### 종료 조건 설계

루프 엔지니어링의 절반은 **언제 멈출지** 입니다.

- ✅ **목표 달성**: 명시적 성공 기준(테스트 통과, 검증 에이전트 승인)
- ✅ **예산 한계**: 최대 토큰/스텝/시간 — 초과 시 강제 종료(무한 루프 방지)
- ✅ **교착 감지**: 같은 행동 반복, 진전 없음 → 에스컬레이션 또는 중단
- ⚠️ **조기 종료 주의**: 모델이 "다 됐다"고 단정하기 쉬움 → 검증 단계로 사실 확인

## 단일 호출 vs 에이전트

모든 문제에 루프가 필요하지는 않습니다.

- **단일 호출(워크플로)**: 단계가 고정·예측 가능 → 결정적 코드로 연결하는 게 더 안정적·저렴
- **에이전트(루프)**: 단계 수·순서가 입력에 따라 달라짐 → 모델이 동적으로 결정

> 경험칙: **결정적으로 짤 수 있으면 코드로, 그렇지 못한 부분만 모델에게** 맡긴다.
