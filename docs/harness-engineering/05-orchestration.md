# 05. 오케스트레이션

단일 에이전트 루프로 부족할 때, **여러 에이전트·단계를 구조적으로 엮어** 더 넓고 깊은 작업을 처리합니다.

## 언제 멀티 에이전트인가

| 동기 | 패턴 |
| --- | --- |
| **포괄성** — 한 컨텍스트가 담기 어려운 넓은 범위 | 분해 후 병렬 팬아웃 |
| **확신** — 결과의 신뢰도를 높이고 싶다 | 독립 관점 + 적대적 검증 |
| **규모** — 마이그레이션/감사처럼 양이 큰 작업 | 항목별 파이프라인 |

> 모든 작업에 멀티 에이전트가 필요하진 않다. **단순/이미 검증된 작업은 단일 에이전트**가 더 싸고 빠르다.

## 서브에이전트와 위임

- **컨텍스트 격리**: 서브에이전트는 자기만의 컨텍스트에서 작업하고 **결론만** 부모에게 반환 → 부모 컨텍스트 절약
- **전문화**: 탐색/리뷰/실행 등 역할별 에이전트에 맞춤 프롬프트·도구 부여
- **반환 계약**: 서브에이전트의 최종 출력은 사람용 메시지가 아니라 **데이터**로 설계(구조화 출력 권장)

```
부모 에이전트
  ├─ explore 에이전트   → 관련 파일 목록 반환
  ├─ executor 에이전트  → 변경 수행
  └─ reviewer 에이전트  → 변경 검증 (별도 컨텍스트에서)
```

## 핵심 패턴

### 1. 병렬 팬아웃 (Parallel Fan-out)
독립적인 하위 작업을 동시에 실행해 벽시계 시간을 단축. **배리어**에서 모든 결과를 모은다.
- 적합: 서로 의존 없는 N개 작업(여러 파일 리뷰, 다각도 검색)
- 주의: 동시성 상한, 부분 실패 처리(실패 항목은 걸러내기)

### 2. 파이프라인 (Pipeline)
각 항목을 여러 단계에 **독립적으로** 통과시킨다. 항목 A가 3단계일 때 항목 B는 1단계일 수 있음(배리어 없음).
- 적합: 항목별 다단계 처리(발견→변환→검증)
- 장점: 단계 간 동기화 대기 없음 → 벽시계 = 가장 느린 단일 체인

### 3. 적대적 검증 (Adversarial Verify)
발견/주장마다 독립 검증자 N명이 **반박을 시도**. 다수가 반박하면 폐기.
- 그럴듯하지만 틀린 결과를 걸러냄
- 관점 다양화(정확성/보안/재현성)가 동일 검증 반복보다 강함

### 4. 판정 패널 (Judge Panel)
서로 다른 각도로 N개 후보 해법을 생성 → 병렬 심사로 점수화 → 우승안 기반 합성.
- 해 공간이 넓은 설계 문제에 적합

### 5. 루프-언틸-드라이 (Loop Until Dry)
크기를 모르는 발견 작업에서 **K회 연속 새 발견이 없을 때까지** 반복.
- 단순 "N개 찾으면 종료"가 놓치는 꼬리(tail)를 잡음

## 배리어 vs 파이프라인 판단

- **배리어(전체 동기화)가 옳은 경우**: 다음 단계가 이전 단계의 **전체 결과**를 필요로 할 때(전역 중복 제거, 0건이면 조기 종료, "다른 결과와 비교")
- **그 외 기본은 파이프라인**: "그냥 평탄화/필터가 필요해서"는 배리어 사유가 아니다 — 단계 안에서 처리하라

## 워크플로 vs 에이전트 오케스트레이션

- **워크플로(결정적)**: 제어 흐름(루프/조건/팬아웃)을 코드로 고정. 재현성·디버깅 유리.
- **모델 주도 오케스트레이션**: 다음 단계를 모델이 판단. 유연하지만 비결정적.
- 혼합이 흔함: 바깥 골격은 결정적 워크플로, 각 노드 안은 모델 에이전트.

## 비용·실패 관리

- 멀티 에이전트는 토큰을 많이 쓴다 → **규모를 작업 난도에 맞춰라**
- 부분 실패를 전제로 설계: 실패 항목은 null로 떨어뜨리고 나머지는 진행
- 무한 팬아웃 방지: 총 에이전트 수·동시성 상한, 진전 없는 라운드 감지
